SECTION 2: PORTS, WEB ARCHITECTURES, & CHANNELS

2.1 Public & Authenticated Web Environments

  • Corporate Web Gateway: https://www.rotech.com serves as the primary external presentation tier. Triage is limited to content delivery network (CDN) accessibility checks and verifying regional referral locator tools.
  • The Rotech Patient Portal: Managed via Salesforce Experience Cloud, this secure system allows patients to manage their home medical equipment orders, update supply timelines, and handle out-of-pocket invoice balances.
  • Physician Referral Portal: An encrypted clinician portal designed for doctors and hospital coordinators to submit incoming oxygen, sleep apnea, or complex medical equipment prescriptions directly into the Salesforce pipeline.

2.2 Internal Collaboration & Real-Time Chat Infrastructure

Rotech utilizes a dual-layer messaging framework for internal operations, divided by functional context:

  • Operational & Collaboration Messaging (Salesforce Chatter / Slack): Directly embedded alongside customer records to allow case tracking, logistics hand-offs, and compliance reviews.
  • Unified Communications Chat (Cisco Webex App): The primary chat network for real-time peer-to-peer and group messaging across clinical and customer service divisions[cite: 1].
    • Support Scope: L1/L2 teams handle configuration issues, single sign-on (SSO) login loops, persistent notification blocks, and room provisioning. They do not handle direct chat history database restores.

SECTION 3: TIER 1 & TIER 2 SUPPORT SCOPE OF WORK

3.1 Statement of Purpose

Rotech Healthcare Inc. utilizes a tiered Managed Service Provider (MSP) framework to handle day-to-day platform variations, clear transactional blockages, and maximize uptime across the Salesforce, MuleSoft, and Webex integrated stack[cite: 1]. Support teams must operate strictly within their defined technical boundaries.

3.2 Tier 1 (L1) Technical Scope

The L1 Help Desk provides first-contact triage and routine system maintenance. Their operational boundaries are confined to standard administrative setup menus and end-user interface screens.

  • Identity & MFA Administration: Manually resolving password lockouts, clearing frozen directory profiles, and forcing software or hardware token re-synchronizations within Salesforce Setup.
  • Downstream Provisioning Checks: Auditing basic platform entrance flags to ensure user changes originating from iCIMS have properly propagated into Active Directory.
  • CTI Telephony & Chat Triage: Resolving localized browser audio device conflicts, hardware headset mappings, and widget panel synchronization errors on the embedded Cisco Webex container. Handling basic Webex App chat login or notification errors.
  • Portal Navigation Assistance: Assisting external clinic administrators or patients with initial portal registration loops and temporary password generation.

3.3 Tier 2 (L2) Technical Scope

The L2 Application Administration team handles non-code system configuration changes, log evaluation, and advanced structural diagnostics.

  • Access Matrix Auditing: Resolving user-escalated "insufficient access" or field-level formatting blocks by deep-diving into profile sharing rules, permission set groups, and role hierarchies.
  • Middleware Logging Diagnostics: Utilizing the MuleSoft Anypoint runtime console to investigate data stream exceptions between Salesforce CRM and the Infor Lawson ERP database[cite: 1]. L2 must isolate payload failures caused by missing parameters (e.g., incomplete clinical metadata or blank ICD-10 medical codes).
  • Experience Cloud Portal Administration: Managing portal user licenses, modifying declarative layout components on the Physician Portal, and adjusting automated notification configurations for patient accounts.
  • Data Integrity & Deduplication: Managing matching rules to identify and merge duplicate physician referral files or corporate customer accounts without breaking historical audit logs or transactional lineages[cite: 1].

4.1 Tier 1 (L1) Diagnostic Steps

  1. Identity Verification: Look up the user's login history in Salesforce. Check if the corporate account is flagged as active in Active Directory. Reset the user's password, unlock the account, and re-sync their active MFA application token.
  2. Environment Isolation: Instruct the end-user (or portal guest) to log completely out of all sessions. Have them open a clean Chrome Incognito window. If the issue disappears, coordinate a full clear of the user's browser cache, cookies, and conflicting temporary web parameters.
  3. Scope Assessment: Cross-reference active incoming ticket queues. Determine if the error is unique to the user's profile, or if multiple external physicians or field workers across the same regional hub are experiencing identical system behavior.
  4. CTI & Chat Verification: Confirm that the browser application has explicit permissions to access the local system headset microphone. For chat sync errors, verify that the user's Webex desktop client application is fully authenticated and running on the local OS layer.

4.2 Tier 2 (L2) Diagnostic Steps

  1. Permission Auditing: Use the "Log In As" feature to replicate the user's view. Verify if the error is caused by missing permission sets, implicit role hierarchy blockages, or field-level data restrictions.
  2. Portal Member Triage: Check if a locked-out physician account has been disabled due to mismatched NPI (National Provider Identifier) data. Re-verify the portal profile and update sharing sets to re-establish secure health document visibility.
  3. MuleSoft Payload Tracing: Locate the transaction ID within MuleSoft Anypoint Monitoring. Extract the failed JSON/XML payload. Identify the structural formatting variance causing the Infor Lawson database to reject the sync transaction[cite: 1]. If necessary, manually correct the data field in Salesforce to re-trigger the automated sync pipeline.

SECTION 5: SERVICE LEVEL AGREEMENTS & ESCALATION

5.1 Tier Support Matrix

Support Layer System Competency Allowable Time Window Mandatory Escalation Gate
Tier 1 (L1) Identity management, device isolation, basic portal navigation, and Webex client/chat configuration. Max 30 minutes inside L1 queue before mandatory routing. Must document browser test results and user's Active Directory status before escalating.
Tier 2 (L2) Permission modifications, dashboard creation, portal license management, record deduplication, and MuleSoft tracking[cite: 1]. Max 2 hours active analysis time per structural issue. Must attach extracted MuleSoft logs, chat error captures, or profile permission matrix audits.
Tier 3 (L3) Core programmatic Apex, custom LWCs, portal architecture changes, and ERP system schema updates. Hotfix specific based on incident severity. Handled by internal platform architects and code development engineers.

5.2 Handback & Escalation Directory

If an operational issue crosses an L2 threshold, it must be updated with all diagnostic logs and routed directly via the following channels:

  • Tier 3 Code Engineering: Route here if the root cause is isolated to custom Apex Trigger runtime faults or broken JavaScript elements within a Lightning Web Component (LWC) deployed to the Physician Portal.
  • Tier 3 Integration Engineering: Route here if the integration failure requires rewriting core MuleSoft mapping logic, re-architecting data endpoints, or patching database structural changes inside Infor Lawson.
  • Security & Corporate Compliance: Route instantly if triage efforts expose potential data leaks, unexpected user access to protected health variables (PHI), or unlogged administrative modifications[cite: 1].
